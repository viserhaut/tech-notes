マイクロサービスが増えるたびにLBが増える問題

  • NodePortはサービスごとに別ポートが必要で管理が煩雑になる
  • LoadBalancerはサービスごとに外部LBが1台必要。マイクロサービスが10個あれば10台のLBが必要になりコストが爆発する
  • IngressはHTTPのホスト名とパスで1台のエントリポイントから複数サービスへルーティングできる
3 / 8

LoadBalancer vs Ingress — NLB vs ALB相当

観点 LoadBalancer Service Ingress
OSI層 L4（TCP/UDP） L7（HTTP/HTTPS）
AWSの相当 NLB（Network LB） ALB（Application LB）
ルーティング IPとポートのみ ホスト名・パスで振り分け
コスト サービスごとにLBが1台 1台のControllerで複数サービス対応
4 / 8

Ingress Controllerがリクエストをルーティングする

  • IngressリソースはルールのYAMLを定義するだけ。実際にリクエストを処理するのはIngress Controller（Nginx等）
  • Minikubeではminikube addons enable ingressでNginx Ingress Controllerを有効化できる
  • /etc/hostsにホスト名を追加してローカルで動作確認する。本番ではDNSに登録する

Ingress Controllerは別途インストールが必要。kubernetesに組み込みではない

Ingress マニフェストとパスルーティング設定

apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: app-ingress annotations: nginx.ingress.kubernetes.io/rewrite-target: / spec: ingressClassName: nginx rules: - host: app.local http: paths: - path: /api pathType: Prefix backend: service: name: api-service port: number: 8080 - path: / pathType: Prefix backend: service: name: web-service port: number: 80
ingressClassName: nginx
どのIngress Controllerを使うかを指定する
