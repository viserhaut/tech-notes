DeploymentはECSサービスに相当するPod管理リソース

  • ECSサービスがタスク定義から複数タスクを起動・維持するように、Deploymentは指定したreplicas数のPodを常に稼働させる
  • Pod障害時は自動で新しいPodを起動し、replicas数を維持する
  • 裏側でReplicaSetを自動生成。Deploymentはそのライフサイクルを管理するラッパー
3 / 9

ReplicaSet直接操作 vs Deployment — なぜDeploymentを使うか

観点 ReplicaSet（直接） Deployment
ローリング更新 手動でPodを差し替える必要あり 自動でローリング更新を実行する
ロールバック 旧ReplicaSetを手動で復元 kubectl rollout undo で1コマンド
更新履歴 追跡できない revisionHistoryLimit で世代管理
推奨度 直接操作は非推奨 本番運用では常にDeploymentを使う

Section 02
ConfigMap・Secretと
API構築
5 / 9

ConfigMap + SecretでAPIの環境変数を外部注入

  • ConfigMapはAPI_URL・LOG_LEVELなど非機密の設定値を管理する
  • SecretはDB_PASSWORD・API_KEYなど機密値を管理。base64エンコードで保存
  • どちらもenvFromでPodに一括注入できる。イメージを変えずに環境差し替えが可能
6 / 9

Deployment + ConfigMap + Secret マニフェスト

apiVersion: apps/v1 kind: Deployment metadata: name: api-server spec: replicas: 3 selector: matchLabels: app: api template: metadata: labels: app: api spec: containers: - name: api image: api-server:v1.0 envFrom: - configMapRef: name: api-config - secretRef: name: api-secret
replicas: 3
常に3Pod稼働を維持。障害時は自動で新Podを起動
selector.matchLabels
ReplicaSetがどのPodを管理するかをラベルで特定
envFrom
ConfigMapとSecretを両方一括で環境変数に注入
7 / 9

ロールアウトとロールバック — kubectl rollout

  • イメージを更新するだけで自動ローリングアップデート。旧Podを順番に置き換えることで無停止デプロイを実現
  • 問題が発生したらkubectl rollout undoで即座に前バージョンに戻せる
  • rollout statusでデプロイ進捗をリアルタイム確認。CI/CDパイプラインから呼び出すことも多い
