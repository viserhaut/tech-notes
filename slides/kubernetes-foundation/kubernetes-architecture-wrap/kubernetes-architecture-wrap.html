ClusterIP / Headless / NodePort / LoadBalancer — 4種比較

Serviceタイプ アクセス元 用途 AWSの相当
ClusterIP クラスタ内のみ マイクロサービス間通信 ALB Target Group（内部）
Headless クラスタ内のみ StatefulSet・DBのPod直接アクセス Route53 Private（Pod IP返却）
NodePort クラスタ外も可 ローカル開発・確認用 （非推奨）
LoadBalancer クラスタ外（外部LB） 本番の外部公開（1サービス） NLB（Network LB）
3 / 9

マイクロサービス増加でLoadBalancerは破綻する

  • LoadBalancer Serviceはサービスごとに外部LBが1台必要。クラウドプロバイダーのLBリソースが必要
  • マイクロサービスが10個あればLBが10台 → コストとIP枯渇が問題になる
  • また、各LBはL4（TCP）のみ対応。パスベースのルーティングができない
4 / 9

IngressがLoadBalancerの代替になる理由

  • Ingress + Ingress Controllerで1台のエントリポイントから複数サービスへL7ルーティングができる
  • AWSではALB Ingress Controllerを使うとIngress定義からALBが自動プロビジョニングされる。1台のALBで複数サービスに対応
  • TLS終端・ホスト名ルーティング・パスルーティングすべてを1つのIngressマニフェストで管理できる

Control Plane + Worker Node 全体構成図

7 / 9

kubeletがPod仕様を実現し、kube-proxyがネットワークを制御

  • kubeletは各Worker Nodeで動くエージェント。kube-api-serverからPod仕様を受け取り、コンテナランタイム（containerd）にコンテナ起動を指示する
  • kube-proxyはServiceが作られるとiptablesルールを書き換え、ClusterIPへのパケットをPod IPに転送する
  • kubectl applyからPod起動まで: kubectl → api-server → etcd → scheduler → kubelet → containerd の順に処理される
