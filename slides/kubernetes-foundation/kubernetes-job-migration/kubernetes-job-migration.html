Section 01
Jobとは何か
2 / 9

Jobは「一度きりの実行」リソース、ECSのrun-taskに相当

  • Deploymentは常時Podを維持、Jobは処理完了で終了
  • AWSでの対応は aws ecs run-task ― 一回だけ流したい処理
  • 完了後もJobオブジェクトが残るため手動削除が必要
3 / 9

Deployment / Job / CronJob — 用途の違い

リソース 動作 向いている用途 AWS対応
Deployment Podを常時維持・再起動 APIサーバ / Webアプリ ECS Service
Job 処理完了で終了、再起動なし DBマイグレーション / バッチ aws ecs run-task
CronJob スケジュール通りにJobを生成 定期レポート / 定期クリーンアップ EventBridge + ECS

Section 02
Minikubeで
イメージを持ち込む
5 / 9

minikube image loadでローカルイメージをMinikubeに転送する

  • Minikubeは独立したVMのため、ホストのDockerイメージを直接参照できない
  • minikube image load でイメージを転送してからPodで指定する
  • ECRを経由せずローカルで完結するため高速に検証できる

Jobマニフェストの読み方

apiVersion: batch/v1 kind: Job metadata: name: db-migrator-job spec: backoffLimit: 1 # 失敗時の最大リトライ数 template: spec: restartPolicy: Never # Pod失敗時に再起動しない containers: - name: db-migrator image: db-migrator:latest env: - name: DB_HOST valueFrom: secretKeyRef: name: postgres-secret key: host - name: DB_PASSWORD valueFrom: secretKeyRef: name: postgres-secret key: password
restartPolicy: Never
Pod内コンテナが失敗してもPod自体を再起動しない。Jobでは Never か OnFailure を使う
backoffLimit: 1
Job全体のリトライ上限。1なら最大2回実行される（初回+1回リトライ）
secretKeyRef
DB接続情報はSecretから注入。YAMLにパスワードを直書きしない
再実行時の注意
完了済みJobは同名で再作成不可。kubectl delete jobしてから再apply
7 / 9

再実行はdelete→apply、CI/CDパイプラインにも組み込める

  • 完了したJobは同名で再apply不可 → delete → apply が基本操作
  • GitHub Actionsでapply→wait→logs→deleteの4ステップで自動化できる
  • 同一VPC内からDB接続が必要な処理にJobは最適
